# Scheduler Runtime Implementation

## Status and authority

This document refines the private runtime implementation authorized by
[ADR 0068](./decisions/0068-typed-async-tasks-actors-and-distribution.md) and the
[concurrency architecture](./23-concurrency-actors-and-distribution.md). It does
not add source syntax, a public scheduling-order promise, detached tasks,
untyped coroutine values, or scheduler objects to HIR/MIR.

The semantic contract remains in ADR 0068. Queue topology, victim selection,
worker parking, batching thresholds, and similar policy remain replaceable
runtime details as long as they satisfy the deterministic tests, GC contract,
bounded-resource rules, and backend equivalence requirements below.

## Research basis

This design was reviewed on 2026-07-14 against primary runtime documentation,
source, and scheduler literature.

### BEAM/ERTS

BEAM supplies several useful lessons without becoming Pop Lang's semantic
model:

- normal scheduler threads have per-scheduler run queues and idle schedulers can
  steal work;
- execution is charged in reductions so a process is scheduled out after
  bounded work instead of relying only on voluntary I/O suspension;
- scheduler-local queues reduce global contention, but process/run-queue state
  transitions require careful synchronization;
- blocking native work is classified away from normal scheduler threads into
  dirty CPU or I/O scheduling domains; and
- priority scheduling adds inversion risks and is not a feature to copy without
  a separate language decision.

Sources:

- [ERTS process scheduling and reductions](https://www.erlang.org/doc/apps/erts/erlang.html)
- [ERTS process-management synchronization lessons](https://www.erlang.org/docs/24/apps/erts/processmanagementoptimizations.html)
- [ERTS run-queue and dirty-scheduler statistics](https://www.erlang.org/doc/apps/erts/erlang.html#statistics_run_queue_lengths_all)
- [ERTS native-function scheduling boundaries](https://www.erlang.org/docs/25/man/erl_nif.html)

Pop adopts bounded-work dispatch, sharded queues, stealable ready work,
separate blocking execution, and scheduler observability. It does not adopt
dynamically typed processes, symbolic registration, selective receive,
user-visible scheduler priorities, or one heap/mailbox model for every task.

### Go

Go's most important contribution is the separation of goroutine, operating-
system thread, and logical processor resource. A logical `P` owns scheduler and
allocator resources, while an `M` binds to a `P` to execute user code. Go also
uses local and global queues, periodic global-queue checks for fairness,
bounded local queues, batch spill/steal, throttled searching workers, network
poll integration, and several kinds of safe point.

Sources:

- [Go runtime scheduler structures](https://go.dev/src/runtime/HACKING)
- [Go runnable selection, fairness, spinning, network polling, and queues](https://go.dev/src/runtime/proc.go)
- [Go safe-point and preemption model](https://go.dev/src/runtime/preempt.go)

Pop adopts the logical-resource/thread separation because it aligns with the
existing scheduler-local TLAB, nursery, root, and ownership model. Pop does not
adopt Go's detached `go` statement, anonymous goroutine lifetime, conservative
asynchronous root scan, unbounded blocked-thread compensation, or scheduler
state as source-visible identity.

### Rust and Tokio

Rust itself deliberately defines inert `Future` polling and wake contracts but
does not define an async runtime. Tokio is therefore implementation evidence,
not a language contract. Its useful scheduler lessons are:

- sharded local queues avoid one globally contended ready queue;
- a global injection queue is still needed for external wakeups and overflow;
- local/global polling intervals are fairness-versus-synchronization controls;
- wakeups must be coalesced so one task cannot occupy the queue repeatedly;
- randomized batch stealing, limited simultaneous searchers, and gradual
  worker notification avoid steal storms;
- bounded per-dispatch operation budgets reduce tail latency when resources
  remain continuously ready; and
- blocking work belongs in an explicit blocking pool rather than heuristic
  replacement of a worker that merely appears slow.

Sources:

- [Rust `Future` polling and wake contract](https://doc.rust-lang.org/std/future/trait.Future.html)
- [Rust async runtime boundaries](https://rust-lang.github.io/async-book/part-guide/async-await.html)
- [Tokio work-stealing scheduler design](https://tokio.rs/blog/2019-10-scheduler)
- [Tokio runtime fairness contract](https://docs.rs/tokio/latest/tokio/runtime/)
- [Tokio cooperative operation budgets](https://tokio.rs/blog/2020-04-preemption)
- [Tokio runtime configuration](https://docs.rs/tokio/latest/tokio/runtime/struct.Builder.html)

Pop adopts the runtime techniques that preserve ADR 0068, but not Rust's
library-selected executor semantics, detached spawn default, erased future
trait objects as a language representation, or an unbounded blocking queue.

### Scheduler literature and deterministic testing

Work stealing is well supported for dynamically generated parallel work, but
the theoretical results do not by themselves prove fairness for continuously
ready server tasks or correctness under weak memory. Chase-Lev establishes the
owner/thief deque shape and the need for a precise linearization and reclamation
argument. Later weak-memory work shows that a plausible atomic translation is
not enough. Cooperative fairness research also distinguishes eventual fair
progress from simple round-robin intuition.

CHESS contributes an important test architecture: record the enabled set and
the selected execution at each scheduling point, replay from the beginning,
fail when the recorded selection is not enabled, and systematically explore
alternative choices under explicit bounds.

Sources:

- [Scheduling multithreaded computations by work stealing](https://doi.org/10.1145/324133.324234)
- [Dynamic circular work-stealing deque](https://doi.org/10.1145/1073970.1073974)
- [Correct and efficient work stealing for weak memory models](https://doi.org/10.1145/2442516.2442524)
- [Provably fair cooperative scheduling](https://doi.org/10.22152/programming-journal.org/2024/8/6)
- [CHESS systematic concurrency testing](https://www.microsoft.com/en-us/research/wp-content/uploads/2016/02/tr-2007-149.pdf)

The first Pop implementation therefore starts with safe sharded queues and
explicit state-machine tests. A lock-free deque is an optimization milestone,
not an architectural prerequisite or a reason to introduce unchecked memory.

## Selected Pop design

### Logical schedulers and workers

`SchedulerId` identifies a logical scheduler resource, not an operating-system
thread and not a source-visible task. One logical scheduler owns:

- one managed ready deque;
- scheduler-local allocation/TLAB and nursery state;
- barrier and root-publication buffers;
- timer/reactor delivery bookkeeping assigned to that scheduler;
- local scheduling counters and bounded telemetry; and
- the current binding to at most one normal worker.

A normal worker is a bounded operating-system thread that temporarily binds to
one logical scheduler while it executes managed work. The binding can change
only through a runtime transition that preserves mutator registration, root
publication, TLAB state, and collector epoch obligations. There are never more
normal workers executing managed code than configured logical schedulers.

This separation prevents a blocked native call or worker lifecycle detail from
becoming GC ownership identity. It also keeps future VM workers compatible with
the same PLRI scheduler/collector contract.

### Task scheduler state

The scheduler stores one typed task control record per retained task. Its
internal state machine distinguishes:

```text
Ready -> Running -> Ready
                 -> Suspended -> Ready
                 -> Completed
                 -> Cancelled
                 -> Panicked
```

Cold `Created` state and exact typed completion storage belong to the later
`Task<T>` runtime transition layer. The ready scheduler accepts only a task
whose owner has performed the start-once transition. Terminal state remains
retained until the owning typed task/group releases it.

The control record has explicit queued, running, notified, cancellation-
requested, and terminal facts. A wake is idempotent while ready. A wake racing
with a poll is retained and forces a later poll if the current poll suspends.
No task is present in a ready queue more than once, and a terminal task is never
polled again.

Task panic is captured as a terminal task outcome. It cannot destroy a worker,
poison scheduler state, or become an actor panic boundary by accident. Actor
containment and task/group propagation remain later semantic layers.

### Bounded queues and admission

Every scheduler resource is bounded by validated configuration:

- logical scheduler and normal worker count;
- retained task-control records;
- each local ready deque;
- global injection queue;
- timers and externally registered event sources;
- blocking-worker count and blocking queue;
- deterministic transcript length and exploration budget; and
- telemetry/event buffers.

Zero capacities are rejected. Exhaustion returns a closed typed runtime
failure before mutating task ownership. No queue silently grows, drops a wake,
creates an untracked detached task, or falls back to one thread per task.

Normal in-scheduler wakeups target the current scheduler's local queue when GC
affinity permits. Wakeups from foreign threads, blocking workers, and event
drivers enter the bounded global injection queue. A configured poll interval
forces workers with local traffic to inspect global injection work so local
ping-pong cannot starve newly submitted or externally woken tasks.

The synchronized correctness implementation selects injection work already
owned by the polling logical scheduler before requesting migration. If no
runtime transition hook is installed, migration is disabled and owner-matched
work remains executable without repeatedly removing and reinserting it.

### Fairness and bounded work

Pop uses cooperative scheduling at compiler/runtime safe points, but it does
not rely on application authors remembering to yield. Each dispatch receives a
deterministic work budget. The compiler and trusted runtime decrement it at:

- loop backedges after bounded work;
- compiler-inserted straight-line polls;
- await, channel, actor, timer, and explicit-yield operations;
- allocation slow paths and collector handshakes; and
- accepted native/runtime transitions.

Budget exhaustion returns the running task to the ready tail after publishing
the required frame and GC state. Continuously ready runtime resources must
respect the same budget instead of returning immediate readiness forever.

The required property is bounded eventual progress under the documented
finite-task and bounded-poll assumptions. Pop does not promise FIFO wall-clock
execution or equal shares between independent tasks. Fairness tests use
scheduler decisions/work units, never sleep duration.

The first release has no source-visible task priority. Internal collector,
timer, cancellation, and shutdown work has separate bounded service rules
rather than user-controlled priority classes.

### Work stealing and migration

An idle worker first checks local work, due event/timer work, and global
injection work. It may then enter the searching state. At most a bounded
fraction of normal workers search simultaneously. Victim selection begins at a
per-worker pseudo-randomized offset and covers each eligible peer at most once
per search round.

Successful stealing transfers a bounded batch rather than repeatedly taking
one task. Batch size is capped by the thief's free queue capacity and the
victim's eligible population. A successful search gradually wakes at most one
additional parked worker, avoiding a wake/steal storm.

The first correctness implementation uses one shared park set rather than
targeted per-scheduler park handles. It broadcasts after publishing work,
because an arbitrary single wake could select a worker that cannot execute
scheduler-affine work while the owner remains parked. Targeted gradual wakeup
is an optimization gate that must retain the same lost-wakeup and affinity
tests. A stealing worker defers this notification until it has left the
lost-wakeup check and released the park gate.

A task is stealable only if its suspended frame and captured state satisfy the
accepted GC sharing/migration proof. Otherwise it remains scheduler-affine.
The runtime migration hook performs publication, promotion, isolated-region
transfer, or refusal before the task becomes runnable on the new scheduler.
Queue movement alone never changes object ownership.

The correctness baseline uses Rust standard-library synchronization and no
unchecked memory. Replacing it with a Chase-Lev-style queue requires:

- a written linearization and buffer-reclamation argument;
- weak-memory model tests for push, pop, steal, resize, wake, and shutdown;
- unchanged deterministic scheduler semantics;
- stress tests under task and collector transitions; and
- benchmark evidence that both throughput and tail latency improve.

### Parking, reactor work, and blocking operations

A worker parks only after a lost-wakeup-safe sequence confirms no local,
global, event, timer, GC, or stealable work. Publishing work and notifying a
worker form one synchronization protocol; notification cannot be merely a
performance hint if omitting it can hide already-published work indefinitely.

One bounded event-driver role polls host I/O readiness at a time to avoid kernel
poll contention. Event delivery only marks exact registered operations ready
and injects their owning tasks. Timers use the same wake protocol. The
deterministic scheduler replaces both with virtual time and injected event
readiness.

`Blocks` work never executes on a normal scheduler worker. It enters an
explicit bounded blocking pool with backpressure and a declared shutdown
owner. The runtime does not guess that a slow task is blocking and create an
unbounded replacement thread. A later FFI/library contract may classify CPU-
bound and wait-bound lanes more precisely; until then the implementation must
not invent a source-visible classification.

### GC and runtime transitions

Every normal worker binding has one typed mutator registration. Scheduler loop,
task dispatch, suspend, resume, park, unpark, native entry/exit, and worker
shutdown paths call the collector transition contract explicitly.

[ADR 0072](./decisions/0072-scheduler-mutator-and-task-root-binding.md)
defines the exact worker-mutator and retained task-frame root lifecycle. Every
non-running retained task, including a ready task waiting in any queue, owns one
collector-visible precise root container. Admission and nonterminal poll return
publish that container before queue/state visibility. Dispatch restores any
relocated `RootSlot` values before polling and transfers root responsibility to
the active managed stack. Trusted root-free host tasks use an explicit empty
frame publication; an omitted publication is never interpreted as empty.

Before suspension, the coroutine frame publishes its exact live root map and
resume state. A suspended frame stays precisely scannable without running. A
major-epoch request prevents a worker from beginning another user dispatch
until its current managed state is acknowledged. A worker in bounded foreign
code remains a blocking handshake participant under the existing collector
state model.

Stealing and task migration are delayed when the collector cannot safely move
the associated scheduler-local graph. Scheduler code cannot call back into
parsing, typing, HIR, MIR construction, or compile-time evaluation to recover
missing ownership facts.

The initial correctness stage stores the exact `StackMap` plus one private
strong collector handle for each present root slot. The registry has explicit
container and logical-slot bounds, restores relocated values only for an exact
map and scheduler owner, and keeps the last valid container on refusal. The
stable-generational native facade exposes the same lifecycle. Cross-scheduler
container transfer currently accepts only rootless frames or roots already in
shared ownership; scheduler-local graphs fail closed until an explicit graph
ownership transition is available.

### Deterministic scheduler

The deterministic scheduler is a separate implementation of the same semantic
transition interface. It executes one task poll at a time and records at each
choice point:

- the monotonically increasing decision number;
- the complete ordered set of enabled task/operation identities;
- the selected identity and virtual scheduler;
- the reason each newly enabled operation became ready;
- cancellation, timer, event, channel/select, and actor-mailbox choices; and
- the virtual work count and collector transition epoch.

Replay fails closed when the selected identity is absent, the enabled set
differs, a decision is missing/extra, or a runtime transition has a different
typed identity. Record/replay never consults wall clock, ambient randomness,
network, filesystem, process state, or hash iteration order.

Bounded systematic exploration derives alternative transcripts from recorded
enabled sets. It prioritizes small numbers of changed decisions and preemptions
before broader exploration, following the useful CHESS testing discipline
without making its host-thread model a Pop semantic dependency.

### Observability

Versioned scheduler telemetry records at least:

- retained, ready, running, suspended, and terminal task counts;
- local/global/blocking queue depth and high-water marks;
- polls, work-budget exhaustion, wake requests, and coalesced wakes;
- steal searches, victims examined, successes, failures, and batch sizes;
- scheduler migrations and GC-delayed migrations;
- worker starts, binds, parks, unparks, and shutdowns;
- event/timer polls and delivery delay in scheduler work units;
- blocking submissions, saturation, execution, and shutdown delay; and
- ready-to-run delay percentiles with the target/runtime profile recorded.

One scheduler observation work unit advances after each successful ready-queue
publication and after each dispatch has restored its exact frame roots. A
ready-to-run sample is the difference between those two monotonic transition
numbers. Rejected publications, stale queue entries, and rejected dispatches do
not create samples. P50, P95, P99, P99.9, and exact maximum delay come from a
fixed-size logarithmic histogram; telemetry never retains an unbounded sample
stream. Benchmark records name the target and runtime profile associated with
the snapshot.

Ready-queue depth counters are aggregate current entries across all logical
scheduler-local queues or the one external injection queue, as named. Their
high-water marks are the maximum aggregate depth observed after an atomic queue
publication, not the maximum depth of an unspecified individual local queue.
Blocking queue depth excludes currently executing blocking operations; those
have a distinct current/high-water counter.

Hot paths update sharded counters. A consistent global snapshot may cost more
and is not performed implicitly by normal scheduling. Telemetry contains
runtime facts only and does not enable task/frame reflection.

## Test and benchmark gates

Implementation order remains architecture, failing tests, then code.

### Deterministic correctness

- valid and invalid bounded configuration;
- exact ready/running/suspended/terminal transitions;
- start-once handoff and no poll after terminal completion;
- duplicate-wake coalescing and wake-during-poll retention;
- cooperative cancellation and cancellation-mask integration points;
- panic isolation without worker loss or poisoned scheduler state;
- terminal-state retention and explicit capacity release;
- bounded fairness for always-ready, ping-pong, and external-injection tasks;
- record/replay success and enabled-set mismatch rejection;
- virtual timer/event ordering and bounded systematic exploration;
- scheduler-affine work never stolen; and
- migration only after the exact collector hook succeeds.

### Concurrent correctness and stress

- simultaneous wake, poll, cancellation, release, and shutdown races;
- queue-full admission without lost or duplicated work;
- bounded batch stealing under one busy and many idle schedulers;
- no steal storm when many workers become idle together;
- lost-wakeup-free park/publication races;
- mutator registration, epoch acknowledgement, root publication, and worker
  binding under GC pressure;
- local nursery collection while other schedulers continue;
- blocking-pool saturation without normal-worker starvation; and
- clean shutdown with suspended, externally waking, panicked, and blocking
  work.

Every concurrency failure must emit a replayable seed/transcript or a bounded
state-transition trace sufficient to reproduce it.

### Performance and scale

The versioned benchmark suite measures:

- cold task-control creation and retained terminal release;
- ready-task polls per second at 1, 2, 4, and available-core workers;
- local wake, foreign wake, global injection, and park/unpark latency;
- balanced queues, one hot queue, burst arrival, ping-pong, and steal storms;
- one million suspended minimal frames on a declared memory profile;
- timer/event fan-out and continuously ready I/O fairness;
- blocking-pool saturation;
- scheduler plus minor/major GC interaction; and
- P50, P95, P99, P99.9, maximum delay, CPU, committed memory, context
  switches, steals, and queue high-water marks.

The synchronized-reference harness emits one tab-separated key/value record per
sample. `schema=pop-scheduler-benchmark-v1` established target, build and
scheduler stage, declared host profile, workload, worker count, logical
operation count, checksum, whole-workload time, initial dispatch latency, and
the first typed task/event counters. `schema=pop-scheduler-benchmark-v2`
supersedes it by adding current/high-water local, injection, and blocking queue
depths, active blocking work, bounded steal-search outcomes and batch size, and
worker lifecycle counters. Historical `v1` records retain their original
meaning; the current harness emits `v2` and consumers fail closed on an unknown
schema identity.

A latency field must name its measurement scope; initial dispatch latency must
not be presented as timer, event, cancellation, or GC pause latency. Profiles
that exercise one million suspended frames or collect operating-system CPU,
memory, context-switch, and GC interaction facts remain separately labelled
profiles rather than implied properties of the default local run.

No lock-free, handoff-slot, batch-size, search-throttle, or polling-interval
optimization lands on throughput alone. It must preserve checksums/transcripts,
not regress the paired latency/throughput workloads beyond the declared gate,
and retain the same GC and cancellation behavior.

## Delivery sequence

1. Land the deterministic task-control state machine and enabled-set
   record/replay tests.
2. Land the bounded synchronized M:N correctness scheduler with local/global
   queues, wake coalescing, parking, panic containment, and telemetry.
3. Integrate logical scheduler/worker bindings with mutator registration,
   precise suspended-frame roots, collector epochs, and migration refusal.
4. Add virtual and host timer/event drivers plus the bounded blocking pool.
5. Add systematic schedule exploration, concurrent stress, race/sanitizer
   coverage, and the versioned scheduler benchmark suite.
6. Optimize queue and wake paths only from measured evidence, retaining the
   synchronized implementation as a differential/reference scheduler until the
   replacement has equivalent proof.

Async syntax, HIR/MIR coroutine frames, `Task<T>`, channels, actors, and cluster
work build on these gates; they do not weaken or bypass them.

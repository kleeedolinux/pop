# Public Library Implementation and Migration Plan

## Delivery rule

Architecture placement does not stabilize an API. Each implementation slice
follows architecture, failing conformance tests, minimal implementation, API and
cost review, then documentation. A phase may overlap in research, but its public
surface cannot stabilize before its prerequisites and definition of done.

## Review gates per API family

Each family proposal includes:

1. Package/Bubble identity, public roots/subnamespaces, tier, dependencies, and
   target capability manifest.
2. Three call sites: concise default, advanced configuration, and efficient
   stream/view/buffer/resource use.
3. Complete public types/functions/errors with no placeholder dynamic values.
4. Allocation, ownership, copying, iteration, dispatch, blocking/suspension,
   native-boundary, and asymptotic-cost table.
5. Positive, rejection, limits, cleanup, cancellation, portability, security,
   and cross-backend test matrix.
6. Benchmarks that measure any proposed performance promise; intended but
   unmeasured goals remain labeled as such.
7. Checked XML documentation, compiled examples, migration notes, and an API
   baseline classified as experimental or stable.

## First implementation pull requests

These slices establish reusable conventions without pretending to deliver the
whole ecosystem:

1. **Publish the library manifest schema.** Add tier/status/capability/cost
   metadata and machine checks for the root inventory. No new runtime API.
2. **Establish the modular source-library build.** Compile ordinary `.pop`
   Modules for `Pop.Standard` by API-family ownership against verified
   `Pop.Internal` references, emit one reserved Bubble artifact, and prove that
   adding a portable function does not change compiler, HIR, MIR, or backend
   code. Keep the Rust bootstrap modules as focused adapters until their
   portable bodies migrate. Land discovery plus verified HIR/MIR conformance
   first; artifact emission/loading completes this slice rather than being
   implied by source validation alone.
3. **Bootstrap the official extension Packages.** Add independent manifests,
   versions, dependency graphs, namespace roots, and builds for `Pop.Data`,
   `Pop.Ai`, `Pop.Cli`, `Pop.Rpc`, `Pop.Syntax`, and `Pop.Lsp`.
4. **Freeze the bootstrap prelude baseline.** Inventory `print`, primitive and
   protocol identities; decide which roots, if any, are implicitly available.
5. **Establish `Bytes` views and reusable buffers.** Define ownership, bounds,
   growth, allocation fixtures, HIR/MIR representation, and interpreter/LLVM
   differential tests.
6. **Establish `Text.View` and Unicode-safe slicing.** Build on Bytes, pin the
   Unicode data version, and replace ambiguous bootstrap slicing contracts.
7. **Establish lazy `Sequence` adapters.** Prove no hidden materialization,
   short-circuit behavior, and exact iterator protocol identities.
8. **Establish `Codec` adapter identities plus bounded `Json.decode`.** Use a
   small typed record schema and direct decode path; defer general code
   generation until the metadata proposal is complete.
9. **Establish `Path` lexical values.** Keep filesystem authority out of the
   first slice; test target-independent normalization and native encoding facts.
10. **Establish `Time.Duration` and injected `Clock`.** Separate pure values from
   PLRI clocks and create deterministic test-time conventions.

Each is independently reviewable and leaves the workspace passing. `File`,
async tasks, network, and cryptography follow only after the relevant resource,
effect, and PLRI decisions are accepted.

## Current implementation inventory and migration

| Current surface | Status | Required action |
| --- | --- | --- |
| modular `Pop.Internal`/`Pop.Standard` source roots | HIR/MIR source validation and logical typed public-function metadata consumption implemented | encode/load verified on-disk `.poplib` artifacts, link referenced implementations, and migrate focused portable bodies |
| `print` bootstrap function | implemented bootstrap | retain provisionally; specify terminal/stream, encoding, allocation, and error behavior before stabilization |
| checked arithmetic helpers | prototype | map into `Math`; keep exact overflow semantics and add API/cost baseline |
| text slicing helpers | prototype | replace byte ambiguity with explicit `Text.View`/byte operations; migration diagnostic if spelling changes |
| sequence helpers | prototype | retain only after lazy/materializing behavior and allocation are tested |
| prelude protocols/types | implemented bootstrap metadata | snapshot exact identities; additions require compatibility and collision review |
| trusted attributes | implemented bootstrap metadata | keep compiler roles; public metadata families require separate generated-adapter work |

There is no released broad standard API to preserve. Compatibility is not
claimed for planned names. Once a public API is stabilized, renames require a
deprecation edition, exact semantic replacement, optional automated rewrite,
and documentation migration. An alias is rejected when it widens the prelude,
keeps a verbose model alive, or obscures a cost/security change.

## Documentation ownership

The owning Package maintains its catalog row, checked API documentation,
examples, cost table, capability/security notes, target matrix, API snapshot,
benchmarks, and migration guide. The architecture index owns root placement and
tier rules. Tooling renders status prominently so planned and implemented APIs
cannot be confused.

## Unresolved dependencies

- Accepted syntax and MIR semantics for async functions, awaiting, cancellation,
  and suspension-capable cleanup. Synchronous result propagation and lexical
  cleanup are fixed by ADR 0052.
- Borrow/view lifetime rules sufficient for non-owning text, bytes, frames, and
  native buffers.
- Generic specialization and code-size policy.
- Public effect/cost metadata schema and which facts are compiler-checked.
- Stable public compiler/source schemas for tooling Packages.
- Initial supported operating systems, architectures, and WebAssembly/web-host
  capability profile.
- Official Package signing, codec licensing metadata, and security response
  policy.
- Exact prelude root list; no catalog root is implicitly imported merely by
  appearing in this architecture.

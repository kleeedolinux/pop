# Closed Design Questions

- Status: accepted architecture baseline
- Date: 2026-07-10

This document answers every question in
[Open design questions](./08-open-design-questions.md). Each answer is binding
for initial implementation unless a focused ADR supersedes it.

## Language semantics

### 1. Numeric types

Pop Lang has explicit fixed-width numeric types:

- `Int8`, `Int16`, `Int32`, `Int64`;
- `UInt8`, `UInt16`, `UInt32`, `UInt64`;
- `Float32`, `Float64`.

`Int` is a platform-independent alias for `Int64`, not pointer-sized. `Float` is
an alias for `Float64`. `Byte` aliases `UInt8`. Target-sized integers, if later
needed for FFI/layout, use explicit `index`/ABI-specific types and do not infect
ordinary arithmetic.

Integer overflow traps in checked/default arithmetic. Explicit wrapping methods
and MIR operations provide modular arithmetic. Division by zero traps. Floating-
point behavior follows IEEE 754 with a later specification for reproducibility-
sensitive constant folding.

Decimal-point or base-ten-exponent literals have an expected float format or
default to `Float64`. Explicit numeric casts use target-type call syntax such as
`Float64(value)`. Integer targets are range-checked, float-to-integer truncates
toward zero, and invalid checked casts trap with `NumericConversion`. Numeric
ordering includes `<`, `<=`, `>`, and `>=`, with ordered IEEE NaN behavior. See
ADR 0040.

String concatenation uses Luau's `..`, and backtick interpolation accepts only
statically known `String`, `Boolean`, and numeric primitives. `String(value)` is
the explicit locale-independent primitive formatting form. There is no
universal `toString`, implicit call conversion, runtime type inspection, or
dynamic formatting fallback. Literal escapes and exact formatting are fixed by
ADR 0041.

### 2. Nil and optionality

`nil` exists, but only `T?`/`T | nil` can contain it. Non-optional references,
fields, parameters, and results cannot be `nil`. Flow checks narrow optionals.
There is no ambient nullable mode.

### 3. Class inheritance

Classes support single implementation inheritance and multiple interface
implementation. Classes are sealed by default; `open class` explicitly permits
subclassing, and overridable methods are explicit. Composition is preferred and
does not require special syntax.

This C#-like nominal shape keeps one object-layout ancestry while avoiding
accidental fragile hierarchies.

### 4. Interfaces

Interfaces are nominal for the initial language. A class explicitly declares
the interfaces it implements, and the compiler verifies the contract.
Structural records remain structural data but do not implicitly implement
behavioral interfaces.

### 5. Operator customization

There are no user metatables/metamethods. Operators map to a closed set of
compiler-known nominal interfaces such as `Add<T>`, `Equal<T>`, `Order<T>`, and
`Iterable<T>`. Operator declarations must be visible, typed, coherent, and
cannot be added retroactively to two foreign types in one library.

### 6. Multiple returns

Multiple returns are Luau-like syntax over fully static type packs. A fixed pack
lowers to a tuple-like MIR result. A variadic tail has a known repeated type or
generic pack parameter. There is no dynamic variadic result. Fixed packs use a
parenthesized result annotation, comma returns, and exact-arity multiple local
declaration/assignment. There is no implicit `nil` padding or discarded extra
value. See ADR 0045.

### 6.1 Numeric ranges and loop control

The first `for` form is Luau's inclusive numeric clause: `for index = first,
last, step do`. Bounds and the optional step have one fixed integer type and are
evaluated once; the binding is immutable. `break` and `continue` target the
innermost loop. Generalized `for value in source do` uses the exact reserved
`Iterable<T>`/`Iterator<T>` protocols and `Iteration<T>` step union; tuple items
support exact Luau-shaped multiple bindings. It performs no implicit disposal.
See ADR 0042 and ADR 0053.

The corresponding first-class immutable value is
`Range.create(first, last, step?)`. It accepts one exact fixed integer type,
implements `Iterable<TInteger>`, and preserves the same inclusive progression,
zero-step trap, and checked advancement. It does not add a range operator or
change the direct numeric `for` clause. See ADR 0056.

### 6.2 Conditional values

`if condition then value else alternative` is a lazy expression with an exact
`Boolean` condition and one statically common branch type. Statement conditionals
use Luau's `elseif` keyword and one final `end`. There is no truthiness, `?:`,
dynamic common type, or implicit union construction. See ADR 0043.

### 6.3 Compound assignment

`+=`, `-=`, `*=`, `/=`, and `%=` apply the corresponding typed numeric
operation; `..=` applies typed `String` concatenation. Mutable locals/captures,
declared class fields, and array elements are accepted targets. A receiver and
index evaluate exactly once before the current value and right-hand side, and
indexed compound reads trap rather than producing an optional. Unsupported
underlying operators do not gain compound spellings implicitly. See ADR 0044.

### 7. Equality, identity, and hashing

- Primitive equality is value equality with precisely specified numeric rules.
- Strings compare by contents even if interned internally.
- Records and tuples support value equality when every element does.
- Classes use identity equality by default; a class can implement the explicit
  typed equality protocol.
- Arrays, maps, tables, functions, and coroutines are not structurally equal by
  default; identity comparison is explicit where supported.
- Hashability is a separate interface and must be consistent with equality.
- Floating-point NaN and signed-zero hashing receive one canonical rule before
  the collection milestone.

### 8. Mutable globals and module cycles

Module constants are preferred. Mutable module state is allowed only through an
explicit module-level declaration and is private unless exposed through typed
functions. It is initialized in declared source order after dependencies.

Runtime initialization cycles are compile-time errors. Type-only cycles are
allowed. A future explicit lazy value can break a cycle without weakening the
module loader.

Namespace constants used by runtime functions are substituted from their
single deterministic evaluated value. They do not allocate a mutable global or
introduce runtime name lookup. The initial runtime-usable set is primitive
values plus recursively fixed tuples. See ADR 0047.

## Types and generics

Namespace `type Name = Target` aliases erase to the target semantic type before
HIR. The initial form is non-generic, permits acyclic alias chains, and creates
no nominal or runtime identity. See ADR 0048.

Enums are nominal payload-free case sets with stable `EnumCaseId` identities
and compact zero-based `UInt32` discriminants. They support exact same-enum
equality without implicit integer conversion and remain distinct from tagged
unions. See ADR 0049.

### 1. Generic representation

Generics use a hybrid lowering:

- specialize value types and performance-critical concrete instantiations;
- share code through typed dictionaries/witness tables for compatible reference
  representations;
- allow profile/optimization-guided extra specialization without changing
  semantics.

HIR records semantic generic arguments; MIR lowering owns the strategy.

The bootstrap executable subset accepts explicit type arguments on generic
functions and tagged-union cases plus expected-type construction of generic
records. The complete first-release boundary adds one nominal colon bound,
whole-call type-argument inference, generic classes/interfaces, and verified
portable cross-Bubble specialization capsules. MIR fully specializes reachable
concrete instances and deduplicates equivalent instantiations. Typed dictionary
sharing is an optional verified MIR optimization, not a `0.1.0` requirement or
a semantic fallback. See ADR 0050 and ADR 0054.

### 2. Structural record subtyping

Record field bindings are immutable in the initial language and allow width
subtyping. Typed `with` creates updated values; contained collections retain
their own explicit mutability. Identity/invariant-heavy mutable state uses
classes. If mutable record fields are ever added, their field set/types must be
invariant.

### 3. Mutable collection variance

Mutable arrays, maps, and tables are invariant. Read-only interfaces may be
covariant in produced values; write-only interfaces may be contravariant where
the type system can prove soundness. No variance is inferred from naming alone.

Arrays are fixed-length, one-based collections. Bulk construction requires an
explicit initial value; indexed writes and checked reads trap out of bounds,
while ordinary indexed reads return an optional. See ADR 0034.

Tables use exact typed keys and optional indexed reads. Indexed assignment
inserts or replaces entries in deterministic insertion order and may grow the
table without changing its identity or invariant type. Assigning `nil` is not a
deletion operation. See ADR 0046.

### 4. Flow narrowing lifetime

Narrowing attaches to a versioned place. It survives pure calls that cannot
mutate or alias the place. It is invalidated by relevant writes, escaping
aliases, unknown-effect calls, captured mutation, FFI/unsafe calls, and
suspension points unless the effect system proves stability.

### 5. Optional flow surface

`== nil` and `~= nil` narrow stable versioned places on their corresponding
edges. `if local value = optional` and `while local value = optional` bind the
present inner value without using truthiness. `optional ?? fallback` lazily
defaults absence, and postfix `optional?` returns `nil` from a
single-optional-result function without relating the two inner types. Postfix
`?` propagates only optional absence. The reserved `Result<T, TError>` union
has `Ok` and `Error` cases; prefix `try` propagates only an exactly matching
error type. Closed `error` declarations define nominal recoverable error
families, and `defer ... end` supplies deterministic lexical last-in, first-out
cleanup. See ADR 0051 and ADR 0052.

### 6. Internal top type

The solver may use an internal `Top` constraint type. It is not source-visible,
has no operations, cannot enter HIR/MIR, and cannot become a fallback when
inference fails.

## Compile time and attributes

### 1. Compile-time function eligibility

Literal/aggregate arithmetic and pure constant expressions evaluate implicitly
when required. A function must be marked `@CompileTime` to be called by UDA,
type-level, or required constant evaluation. The checker validates its complete
effect/call graph. The optimizer may fold other functions without changing
program acceptance.

### 2. Valid UDA values

UDAs contain immutable booleans, numbers, strings, enum cases, resolved type or
symbol handles, tuples, records, and homogeneous arrays composed from those
types. No mutable table, closure state, raw pointer, runtime object, backend
handle, or source string is valid.

### 3. UDA transformation power

Version one UDAs can produce constants, diagnostics, specialization choices,
metadata projections, and instances of compiler-defined typed adapter protocols.
They cannot create arbitrary declarations. A future hygienic typed declaration
builder requires a new ADR; string/token generation remains permanently banned.

### 4. Compile-time effect notation

PascalCase attributes carry the uncommon contract without adding modifier
syntax: `@CompileTime`, `@Pure`, and narrowly named compiler capabilities. The
compiler infers detailed internal effects and verifies declared promises.

### 5. Build inputs

Compile-time code receives no filesystem, environment, clock, random, process,
or network input in version one. Generated build configuration must be passed as
ordinary typed constants by the project system and becomes an explicit cache
dependency.

### 6. Runtime metadata in the first release

There is no general public runtime reflection. Private metadata required for GC,
dispatch, casts, and stack traces is emitted as needed. User-visible metadata is
available only through explicitly generated typed adapters; UDAs are otherwise
erased.

## Runtime

### 1. Memory manager

The initial memory manager is proof-directed and hybrid. Scalars and
scalar-replaced values allocate no managed storage; proven non-escaping arrays,
aggregates, closures, and common-lifetime graphs use activation-owned storage or
compiler-inferred scoped regions. Exact means every static reclamation is proven
safe on every path, not that every allocation must be classified statically.
Missing proof falls back to a precise concurrent generational tracing GC: a
fast moving nursery plus region-based mostly non-moving mature heap, concurrent
marking/sweeping, thread-local allocation, card marking, and compiler stack
maps. See [Static memory management](./24-static-memory-management.md),
[ADR 0085](./decisions/0085-proof-directed-static-reclamation.md), and
[Garbage collector architecture](./15-garbage-collector-architecture.md).

### 2. Finalizers

User finalizers are excluded from version one. They complicate liveness,
ordering, resurrection, latency, and library unloading. Explicit `close`/scope
cleanup handles resources. A narrow runtime-internal cleanup mechanism is not a
language guarantee.

### 3. Strings and bytes

`String` is an immutable valid UTF-8 sequence. Indexing semantics operate on
bytes only through explicit APIs; iteration defaults to Unicode scalar values.
`Bytes`/`{Byte}` is a separate mutable or immutable byte collection type.
Malformed external bytes require explicit decoding policy.

### 4. Error model

Recoverable expected failures use typed `Result<T, E>`-style values with
Luau-light syntax sugar to propagate them. `panic` represents violated
invariants and normally unwinds for cleanup/diagnostics before terminating the
task/application policy boundary. Exceptions are not the ordinary error API.

### 5. Concurrency

Module initialization is synchronized and published once. Ordinary objects are
not implicitly thread-safe. Data races are forbidden; the first implementation
enforces this through documented synchronization and a sanitizer while a fuller
ownership/sendability system is designed. GC and immutable values are thread-
safe. Coroutines can migrate only when their captured state satisfies the
runtime's safe-sharing rules.

### 6. Minimum private runtime metadata

Each traced allocation can reach a compact `TypeInfo` describing pointer layout,
object size/layout class, and dispatch/type identity as required. Side metadata
stores mark/age/card state. Stack maps identify precise roots. Optional name and
line tables support diagnostics and are separable/strippable. No field-name or
UDA registry is required for collection.

## Toolchain and packaging

### Toolchain localization

Human CLI, compiler-diagnostic, and language-server presentation uses private
embedded TOML catalogs for English, Simplified Chinese, Japanese, Brazilian
Portuguese, and neutral Spanish. Selection and fallback follow ADR 0088 and are
immutable per invocation/session. Machine protocols and semantic identities are
locale invariant. This is separate from the planned YAML-based public
`Pop.Locale`/`Pop.Resource` application contract.

### Private semantic editor queries

Before the public `Pop.Syntax` and `Pop.Lsp` schemas stabilize, the private
language server consumes a version-coupled compiler tooling projection. ADR
0089 authorizes compiler diagnostics, checked-documentation hover, and
namespace-scope document symbols for immutable open-Module snapshots. ADR 0090
adds structured related diagnostics, current compiler quick fixes, direct-call
parameter hints, and conservative same-Bubble analysis for dependency-free
Packages. The projection carries typed identities and source spans, never
compiler arenas, HIR/MIR values, CLI text, or string-based resolution.
Completion, signature help, cross-Bubble navigation, references, rename,
formatting, semantic tokens, incremental edits, and complete
Workspace/dependency analysis remain separately reviewed work.

### 1. Compiler implementation language and repository layout

The initial compiler, runtime, and first-party tools use Rust edition 2024 in a
resolver-3 virtual Cargo workspace. Focused crates mirror the ownership
boundaries in the compiler component architecture, and architecture tests reject
forbidden dependency directions. The host implementation does not change Pop
Lang source syntax or its Item/Module/Bubble/Package/Workspace model. See ADR
0018. ADR 0038 refines the runtime into a pure PLRI contract, reusable collector,
native-ABI vocabulary, and native host facade so VM/interpreter composition does
not depend on C exports or process-global state.

### 2. Source extension and command

The source extension is `.pop`. The command-line tool is `pop`. The product and
language name remains **Pop Lang** in prose, titles, and branding.

### 3. Item, Module, Bubble, Package, and Workspace identity

The fixed hierarchy is `Item → Module → Bubble → Package → Workspace`. Modules
are `.pop` files, Bubbles are crate-like compilation/`internal` boundaries,
Packages are directories with `bubble.toml`, and Workspaces group Packages under
one `bubble.lock`. Namespaces organize names inside a Bubble. Paths guide
discovery but are not Package/Bubble semantic identity.

See [CLI, tooling, and units of code](./21-cli-tooling-and-code-units.md) and
[Bubbles, namespaces, artifacts, and loading](./14-libraries-namespaces-and-loading.md).

### 4. Locking and reproducibility

`bubble.toml` declares dependency requirements; generated `bubble.lock`
owns exact versions, sources, content hashes, features, and target selections.
The build graph records compiler edition, PLRI ABI, and explicit generated
configuration inputs.

### 5. Conventional targets and CLI

The default roots are `src/lib.pop`, `src/main.pop`, `src/bin/`, `tests/`,
`examples/`, and `benchmarks/`. The single command is `pop`; full commands use
complete names such as `pop check`, `pop build`, `pop run`, `pop test`,
`pop benchmark`, `pop format`, `pop lint`, `pop documentation`, `pop add`, `pop tree`, and
`pop metadata`. The experimental `pop transpile <source.pop> --to c` bootstrap
form emits deterministic runtime-free C11 from optimized verified MIR. Workspace
selection uses Package/Bubble-aware flags.

For compiler bootstrap and IR inspection only, `pop check <source.pop> --dump
hir|mir|ll` supplies an ephemeral one-Module Bubble inside an ephemeral Package and
Workspace. It is not Package discovery or artifact identity, cannot resolve
dependencies or widen visibility, and makes no stable dump-format promise.
Normal programs continue to use manifest-driven Package/Bubble selection.

The C transpilation experiment requires a no-argument canonical binary entry in
its initial runtime-free slice. It preserves checked scalar semantics without C
undefined behavior and supports only typed integer and literal-string output
through private C standard-I/O adapters. Managed allocation, PLRI, dispatch,
unwind, coroutine, unsafe, and FFI operations are rejected instead of receiving
fallbacks. See
[ADR 0059](./decisions/0059-experimental-secure-c-transpilation-backend.md).

Every binary Bubble, including the standalone native conformance path, resolves
one `main` by `SymbolId`. Entry visibility may be omitted or `private`; its
parameters are either empty or `Array<String>`, and its results are either empty
or `Int`. No-result completion means status zero. Invalid platform UTF-8 traps
only when the entry requests arguments. Library Bubbles require no `main`, and
ordinary functions may have any statically declared result shape. See ADRs 0025
and 0026.

### 6. Serialized MIR and bytecode stability

Textual MIR is a deterministic test/debug format, not a stable public ABI.
Serialized MIR caches are versioned and may be invalidated across compiler
versions. Released VM bytecode uses an explicit format version and verifier but
promises compatibility only within a documented language/runtime release line.
Source/Bubble public API and runtime ABI stability matter more than eternal IR
compatibility.

### 7. Lock and `.poplib` physical encoding

`bubble.lock`, `bubble.manifest`, and `reference.metadata` use bounded canonical
UTF-8 JSON with independent schema versions. Artifact files use normalized
relative paths, recorded sizes, and lowercase hexadecimal SHA-256 inventories.
Portable generic capsules serialize the verified logical HIR/type payload and
fail closed on any version, identity, hash, visibility, type, effect, or budget
mismatch. Target objects stay opaque and documentation stays separate. See ADR
0055.

## Milestone 3 integration decisions

- Local and anonymous functions use Luau-shaped `function ... end` syntax.
  Read-only captures are by value; written captures share typed cells. Closure
  environments are native managed objects (ADR 0019).
- Interfaces contain public instance method signatures; classes explicitly use
  `implements`, and interface dispatch uses resolved nominal slots (ADR 0020).
- The initial tagged-union `match` is an exhaustive statement with
  `when ... then` arms, no wildcard/guard, and resolved case/payload identities
  (ADR 0021).
- Function/call effect summaries are closed; traps, panic unwinding, cleanup
  edges, allocations, maps, roots, barriers, and safe points are explicit in
  MIR/PLRI. The source has no initial effect punctuation: effects are the least
  fixed point of typed operations and resolved calls, including recursive
  components, while indirect calls use their function type's closed summary
  (ADR 0022).
- `Result<T, TError>` is the reserved `Pop.Standard` `Ok`/`Error` union;
  nominal `error` declarations provide stable recoverable cases, prefix `try`
  propagates only an exact error type, and `defer ... end` registers lexical
  last-in, first-out cleanup on every non-trap scope exit (ADR 0052).
- Attribute usage is declared with trusted `@AttributeUsage` and optional
  `@AttributeValidator`; typed `attribute<<A>>`/`hasAttribute<<A>>` queries never
  accept runtime names. A validator exactly mirrors the attribute constructor
  parameters, returns one `Boolean`, and receives normalized canonical
  arguments. Namespace-targeted attributes precede the file's `namespace`
  header; attributes after it attach to the next Item (ADR 0023).

## Naming and syntax decisions added by the revision

- Namespaces, Packages, Bubbles, types, interfaces, enum cases, and attributes use
  `PascalCase`.
- Built-in types also use `PascalCase`: `String`, `Int`, `UInt32`, `Boolean`,
  `Float64`, `Byte`, and `Never`.
- Functions, methods, fields, locals, parameters, modules, and filenames use
  `camelCase`.
- Only constants use `UPPER_SNAKE_CASE`.
- Public/API identifiers use complete words; arbitrary truncations such as
  `Iter`, `Config`, and `Sync` are not canonical. Established technical names
  such as `Json`, `Http`, `Io`, and `Async` remain word-cased.
- File headers use semicolon-free `namespace` and `using` declarations.
- Every namespace-scope declaration resolves to `public`, `internal`, or
  `private`; omission defaults to `internal` except for the private binary-root
  `main` shorthand. There is no `export` keyword/list or re-export operation.
- An omitted function result annotation is an empty result pack, never inferred
  from returned values; parameters remain explicitly typed.
- Functions may live directly in namespaces without static utility classes,
  singleton objects, or module tables.
- `using` changes name resolution only; `bubble.toml`, locked Bubble references,
  and `BubbleContext` load implementations.
- Canonical style is defined in
  [Syntax and nomenclature](./13-syntax-and-nomenclature.md).
- Non-empty XML documentation elements never use an inline source form. Opening,
  content, and closing tags occupy separate `---` lines, with an empty `---`
  between sibling top-level contract elements
  ([ADR 0057](./decisions/0057-multiline-xml-documentation-format.md)).
- The two base libraries are `Pop.Internal` (private/trusted) and `Pop.Standard`
  (native portable public foundation), as defined in [Base libraries](./16-base-libraries.md)
  and [Public standard-library architecture](./22-public-standard-library-architecture.md).
- Pop Lang is procedural/functional/data-oriented by default. Records, unions,
  plain functions, generic algorithms, and composition precede interfaces/
  classes; see [Paradigm and API style](./18-paradigm-and-api-style.md).
- `Pop.Standard` uses a curated prelude and contextual short names. ADRs 0031
  and 0032 define the active roots, shallow call-site rules, and explicit cost
  contracts. ADR 0033 keeps `Pop.Data`, `Pop.Ai`, `Pop.Cli`, `Pop.Rpc`,
  `Pop.Syntax`, and `Pop.Lsp` independently installed. Builders and
  object-shaped clients are not default API patterns.
- ADR 0058 freezes the initial prelude. `Sequence` is its sole namespace root;
  the exact primitive/foundation types, typed `print` overloads, and trusted
  attributes come only from verified bootstrap metadata. Optional values use
  `T?`; there is no duplicate nominal `Option<T>` binding. The versioned API
  baseline distinguishes executable prototypes from stabilized APIs.
- Bootstrap prelude functions use stable `StandardFunctionId` identities. The
  executable output bindings are `print(Int) -> ()` and `print(String) -> ()`;
  static argument types select the exact overload, and HIR/MIR never identify
  either adapter by source or native ABI spelling.
- Ordinary source overloads use ADR 0074 exact non-generic selection. Complete
  parameter type packs distinguish candidates; results, conversions,
  declaration order, and runtime values never participate. Generic overload
  groups and bare overloaded function values remain rejected in this slice.
- Sequence protocols are `Iteration<T>`/`Iterable<T>`/`Iterator<T>` and their
  first algorithms are `Sequence.map`/`Sequence.filter`/`Sequence.fold`/
  `Sequence.collect`; `Iter` is not an API name. See ADR 0053.
- The first additional terminal algorithms are `Sequence.any`, `Sequence.all`,
  and `Sequence.count`. They are eager, single-pass, allocate no storage, and
  preserve checked `Int` overflow. Ambiguous `find(T) -> T?` is not accepted.
  See ADR 0061.
- Sequence inspection and visitation first use explicit fallbacks and predicates:
  `isEmpty`, `firstOr`, `lastOr`, `each`, `none`, and `countWhere` have exact
  traversal and O(1)-storage contracts under ADR 0064. A no-fallback result
  waits for ordinary exhaustive matching of the reserved `Iteration<T>` union.
- Lazy Sequence bounds and composition use `take`, `drop`, `takeWhile`,
  `dropWhile`, and `concat`. Nonpositive counts normalize to zero; predicate
  failure and iterator-acquisition order are fixed by ADR 0066.
- Integer Sequence aggregation uses `sum`, `product`, `minOr`, and `maxOr` with
  checked left-to-right `Int` arithmetic and explicit empty fallbacks. Generic
  numeric aggregation waits for accepted numeric protocols; see ADR 0067.
- Sequence projection and composition adds `findOr`, `indexOr`, `sumBy`,
  `productBy`, `minByOr`, `maxByOr`, `append`, `prepend`, and `scan` under ADR
  0075. Projection is fused, extrema preserve the first tie, and lazy adapters
  allocate exactly one typed iterator state object.
- Indexed and last-match inspection plus first-item reduction add
  `elementAtOr`, `findLastOr`, `indexLastOr`, and `reduceOr` under ADR 0076.
  They are fused single-pass operations, allocate no storage, and preserve
  exact traversal and callback counts.
- The first portable `Math` functions are direct `Int` operations named `min`,
  `max`, `abs`, and `gcd`. They allocate nothing and preserve the existing
  checked `IntegerOverflow` trap. Other numeric families wait for their exact
  overload/protocol and cost contracts. See ADR 0062.
- The next portable integer functions are `sign`, `lcm`, and `coprime`.
  Division precedes multiplication in `lcm`, but every magnitude and result
  remains checked `Int`; see ADR 0065.
- The first API-baseline expansion preserves every ADR 0058 prelude identity,
  keeps `Sequence` as the sole implicit namespace root, and appends ordinary
  `Sequence` and `Math` prototypes without compiler-known function identities.
  See ADR 0063.
- The Rust implementations of the two base-library Bubbles use thin explicit
  module inventories. `pop-standard` modules and tests follow public API-family
  ownership; `pop-internal` modules follow trusted runtime-service ownership.
  Ordinary library algorithms do not require compiler, IR, backend, or runtime
  edits. See
  [ADR 0035](./decisions/0035-modular-base-library-implementation.md).
- Cross-Bubble functions use the typed identity pair `(BubbleId, SymbolId)` and
  closed public reference signatures. Ordinary library functions do not gain
  compiler-known bootstrap IDs; unsupported metadata types fail closed. See
  [ADR 0036](./decisions/0036-typed-cross-bubble-function-references.md).
- ADR 0073 makes the reserved Standard reference implicit in ordinary CLI
  Bubbles and requires package linking to consume target bytes reloaded from a
  verified `.poplib`. Independent artifact caches remain gated on deterministic
  remapping of session-local Bubble IDs rather than filename or namespace
  guesses.
- Rust foundation adapters use the typed `#[poplib(...)]` attribute and an
  explicit descriptor inventory. The attribute verifies a fixed native ABI and
  binds only to accepted Standard metadata or an Internal intrinsic; it cannot
  create Pop declarations, stable semantic IDs, or runtime registrations. See
  [ADR 0037](./decisions/0037-typed-rust-foundation-adapter-attribute.md).
- The Rust runtime uses four focused ownership crates: PLRI interface, portable
  collector, native ABI vocabulary, and native host facade. C symbol spellings
  do not live on backend-neutral `RuntimeOperation`; the collector is reusable
  without native linking or process-global state. See
  [ADR 0038](./decisions/0038-modular-portable-runtime-implementation.md).
- A PLRI managed reference is a current opaque physical token, while Pop object
  identity remains stable. Collecting safe points mutate canonical `RootSlot`
  publications; production GC requires backend plus target relocation
  capability and native ABI major version 2. Bootstrap ABI 1.x cannot silently
  satisfy that profile. See
  [ADR 0039](./decisions/0039-relocating-nursery-root-and-backend-contract.md).
- Scheduler workers register one typed mutator for their lifetime and remain
  detached while idle. Every non-running retained task frame, whether ready or
  suspended, owns a collector-visible precise root container. Dispatch restores
  relocated `RootSlot` values before polling, and migration commits only after
  the collector transfers the exact container/ownership. See
  [ADR 0077](./decisions/0077-scheduler-mutator-and-task-root-binding.md).
- Native ABI 1.11, ABI 1.12, ABI 1.13, ABI 1.14, ABI 1.15, ABI 1.16, ABI 1.17,
  ABI 1.18, and ABI 2.0 coexist as distinct closed descriptors. ABI 1.12 adds
  compiler-created task frames and separates cancellation source authority from
  immutable tokens while retaining ABI 1 stable roots; ABI 1.13 adds balanced
  foreign transitions, ABI 1.14 adds balanced managed-thread attachment, ABI
  1.15 adds handle resolution, ABI 1.16 adds managed FFI buffers, ABI 1.17 adds
  immutable-byte payload borrows, and ABI 1.18 adds typed callback
  registration/entry without changing the earlier descriptors.
  ABI 2
  uses `pop_rt_gc_safe_point_v2`, exact writable root arrays, mandatory
  post-call LLVM reload aliases, and a fixed capability query; the stable
  facade cannot report ABI 2 support. See
  [ADR 0078](./decisions/0078-native-abi-2-writable-root-coexistence.md) and
  [ADR 0079](./decisions/0079-native-task-frame-and-cancellation-abi.md).
- Native ABI 1.15 adds exact strong-root resolution for `Ffi.Handle<T>`. Open,
  get, and close use retain-root, resolve-root, and release-root respectively;
  stale or forged generations fail before returning a managed value. See
  [ADR 0082](./decisions/0082-ffi-abi-storage-and-lexical-borrows.md).
- Native ABI 1.16 implements `Ffi.Buffer<T>` as a relocating managed resource
  with separately owned ABI storage. Bootstrap identities 208 and 209 are the
  singleton `Ffi.NullPointerError` and `Ffi.AllocationError` types. Open keeps
  allocation failure distinct from invariant failure; every other operation
  uses one-based indices and is bounds-, layout-, lifecycle-, and
  borrow-checked. See
  [ADR 0083](./decisions/0083-ffi-resource-state-and-native-buffer-abi.md).
- Canonical MIR uses one validated target FFI layout catalog. Buffer open
  constructs its exact typed `Result` only after the closed runtime outcome;
  borrow exposes only the scoped optional pointer while its opaque native
  generation remains backend-private state keyed by `BorrowRegionId`. See
  [ADR 0084](./decisions/0084-canonical-mir-ffi-buffer-operations.md).
- Canonical FFI layout descriptors have full lowercase SHA-256 fingerprints;
  their nonzero compact `FfiAbiLayoutId` is the first eight digest bytes in
  big-endian order. Full fingerprints and descriptor facts remain in artifact
  metadata, and zero or unequal-fingerprint compact collisions fail closed.
  Trusted `Ffi.C.Layout` identities flow through HIR into the target-selected
  catalog; source `Ffi.Buffer` intrinsics never derive layout from `TypeId`,
  declaration order, spelling, or a backend. See
  [ADR 0086](./decisions/0086-canonical-ffi-layout-identities-and-source-catalogs.md).
- `Ffi.Buffer.withPointer` and immutable-`Bytes` `Ffi.withPin` accept only one
  immediate synchronous closure body. Canonical MIR names its nested body and
  `BorrowRegionId`, verifies borrow provenance through that body, and balances
  cleanup on every exit. Native ABI 1.17 returns only the immutable byte payload
  address, length, and a private runtime token; backends never derive a managed
  payload offset. See
  [ADR 0087](./decisions/0087-scoped-ffi-borrow-bodies-and-bytes-pin-abi.md).
- `pop ffi generate` selects one exact target-owned manifest plan and consumes
  one hashed canonical declarative `.popc` descriptor containing exact layouts,
  signatures, and reviewed pointer policy. Schema 1 emits deterministic
  `bindings.pop`, no-shim `bindings.c`, and `native-bindings.popc`, and publishes
  only into an absent destination or accepts a verified byte-identical no-op.
  It invokes no process and rejects executable descriptor code, headers, raw
  source/flags, inferred pointer safety, unsupported ABI shapes, host fallback,
  and destructive replacement.
  See
  [ADR 0093](./decisions/0093-deterministic-ffi-binding-generation.md).
- `Ffi.withCallback` creates one immediate synchronous function/context pair;
  `Ffi.Callback.open` creates one explicitly closed registered resource. Both
  retain an exact typed capture environment behind a runtime-owned opaque
  context, never a managed address. The first stable policy is serialized,
  non-reentrant, synchronous, and panic-contained; native ABI 1.18 validates
  the callback site, context generation, thread/scheduler policy, and balanced
  callback transition. See
  [ADR 0092](./decisions/0092-typed-ffi-callbacks-and-native-transition-abi.md).
- Shared ownership does not imply immutability. A complete shared graph may be
  frozen atomically into distinct `SharedImmutable` metadata, after which every
  payload mutation fails before barrier or heap changes. Verified MIR may omit
  a runtime barrier only while retaining a closed compiler proof; the first
  proof is a same-block, non-escaping `UnpublishedOwner` allocation fact. No new
  source capability syntax is selected. See
  [ADR 0080](./decisions/0080-shared-immutability-and-barrier-proofs.md).
- Proof-directed static reclamation is inferred per allocation use rather than
  fixed per source type. Verified MIR may elide storage, use an
  activation-owned slot, or use a compiler-inferred scoped region; missing
  lifetime/retention proof uses Pop GC. The source gains no lifetime parameters,
  ownership modifiers, manual `free`, implicit destructors, weak edges, or
  universal reference counting. See
  [ADR 0085](./decisions/0085-proof-directed-static-reclamation.md).
- Stable native interoperability uses ordinary namespace functions with exact
  trusted `Ffi.Link`/`Ffi.Foreign` identities and typed manifest link aliases.
  Link plans are target-specific and hash verified; raw flags, shell command
  substitution, runtime symbol lookup, untyped C varargs, and untracked managed
  pointers are excluded. ABI 1.13 adds balanced, thread-bound foreign
  transitions: blocking calls retain exact roots in `HandlesOnly`, reviewed
  nonblocking calls use `BoundedForeign`, and every single-use transition
  restores and writes back its original root publication. ABI 1.14 attaches the
  native program entry before managed allocation and detaches it after return.
  Scoped pins,
  generation-checked handles, callback transitions, and deterministic generated
  source preserve GC and static typing. See
  [ADR 0081](./decisions/0081-statically-bound-native-ffi.md).
- Foreign pointers never expose an arbitrary Pop object address. The first
  managed pin borrows only immutable `Bytes` payload storage through a
  read-only pointer and exact length. General native arrays/structures use
  bounds-checked, aligned `Ffi.Buffer<T>` ownership; long-lived managed values
  use generation-checked `Ffi.Handle<T>`. Fixed-layout records are marshalled
  into separate ABI storage with canonical target fingerprints. See
  [ADR 0082](./decisions/0082-ffi-abi-storage-and-lexical-borrows.md).
- Compiler diagnostics use stable `POP####` codes, warning waves, and structured
  quick fixes as defined in
  [Diagnostics, warnings, and quick fixes](./17-diagnostics-warnings-and-quick-fixes.md).
- The private language server preserves compiler diagnostic labels, notes,
  categories, warning waves, and current safe source edits; it exposes only
  compiler-proven direct-call parameter hints. It selects the nearest Package
  boundary first, then enables conventional same-Bubble analysis only when that
  Bubble has no unresolved dependencies. Nested Packages never merge. See
  [ADR 0090](./decisions/0090-rich-private-editor-analysis.md).
- `pop new` and `pop initialize` create only validated canonical binary or
  library Package scaffolds. They never rewrite identities, overwrite protected
  entries, initialize version control, or download dependencies. See
  [ADR 0091](./decisions/0091-canonical-package-scaffolding.md).
- Accepted architecture is binding but evolvable through ADRs. Any contradiction
  is a bug; restoring Lua's dynamic/table-centered semantics is a release-
  blocking Lua regression. See
  [Architecture conformance and regression policy](./19-architecture-conformance-and-regression-policy.md).
- XML documentation uses Lua-shaped `---` comments with checked C#-style XML and
  Pop-specific typed error/effect/performance tags; see
  [XML documentation comments](./20-xml-documentation-comments.md).
